# From-Migrations Example

Convert `contentful-migration` scripts into cms-sim schemas and simulate content ŌĆö **no Contentful connection needed**.

This is the **4th workflow**: if your team already has migration files as the source of truth for the content model, you can replay them offline with a mock `Migration` object.

## Usage

```bash
# From the package root:

# Step 1: Convert migration files ŌåÆ schema files
cms-sim from-migrations --migrations=examples/from-migrations/migrations/ --output=examples/from-migrations/schemas/

# Step 2: Simulate content from the extracted schemas
cms-sim --schemas=examples/from-migrations/schemas/ --entries-per-type=5 --open
```

Or as a single pipeline:

```bash
cms-sim from-migrations --migrations=examples/from-migrations/migrations/ --output=examples/from-migrations/schemas/ && \
cms-sim --schemas=examples/from-migrations/schemas/ --entries-per-type=5 --open
```

## What's in here

```
migrations/
  001-create-blog.mjs       ŌåÉ prop-style API: creates author + blogPost
  002-add-categories.mjs    ŌåÉ fluent-chaining API: creates category, edits blogPost
  003-refine-fields.mjs     ŌåÉ editField, changeFieldId, createField on existing CTs
schemas/                    ŌåÉ generated by from-migrations (not checked in)
```

## Migration styles

Both Contentful migration API styles are supported:

### Prop-style (object arguments)

```js
const ct = migration.createContentType('blogPost', {
  name: 'Blog Post',
  displayField: 'title',
});
ct.createField('title', { name: 'Title', type: 'Symbol', required: true });
```

### Fluent-chaining

```js
const ct = migration.createContentType('category');
ct.name('Category');
ct.displayField('title');
ct.createField('title').name('Title').type('Symbol').required(true);
```

## What the mock handles

| Operation | Behavior |
|-----------|----------|
| `createContentType` | Captures content type definition |
| `editContentType` | Accumulates changes on existing CT |
| `deleteContentType` | Removes CT from output |
| `createField` | Adds field to CT |
| `editField` | Updates field props |
| `deleteField` | Removes field |
| `changeFieldId` | Renames field id |
| `moveField` | No-op (order not meaningful for schemas) |
| `transformEntries` | No-op (data transform, schema-only mock) |
| `deriveLinkedEntries` | No-op |

## Multi-file accumulation

Migrations are executed in filename order. Each file receives the same `MigrationMock` instance, so file #2 can `editContentType` what file #1 created ŌĆö just like the real `contentful-migration` tool.
